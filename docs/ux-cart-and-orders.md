# UX-сценарий: Корзина и заказы (Telegram-бот Kircraft)

## 1. Обзор

Документ описывает UX-поток управления корзиной и заказами клиента в Telegram-боте магазина изделий из кожи Kircraft. Главное меню не перегружается: доступ к корзине и заказам реализован через **динамическую нижнюю панель**.

---

## 2. Динамическая нижняя панель (главный экран)

**Экран:** главное меню (main-menu).

**Правила отображения:**

| Сценарий | Условие | Содержимое панели |
|----------|---------|-------------------|
| 1 | Корзина пуста, заказов нет | Панель **не отображается** |
| 2 | В корзине есть товары, заказов нет | Одна кнопка: **🛒 Корзина (N)** |
| 3 | Корзина пуста, есть хотя бы один заказ | Одна кнопка: **📦 Мои заказы** |
| 4 | В корзине есть товары и есть заказы | Две кнопки: **🛒 Корзина (N)** и **📦 Мои заказы** |

**Расположение:** под основным блоком кнопок (Каталог, Коллекции, Новинки, Галерея, Написать мастеру, О мастерской). Отступ сверху (pt-1.5), снизу (pb-4).

**Поведение:**
- **Корзина (N)** → переход на экран корзины.
- **Мои заказы** → переход на экран списка заказов.

**Данные:**
- `cartItems.length` — количество позиций в корзине.
- Наличие заказов — запрос к API/БД: у пользователя есть хотя бы один заказ (любой статус, кроме «удалён»).

---

## 3. Полный UX-flow (переходы между экранами)

```
[Главное меню]
    │
    ├─ 🛒 Корзина (N) ──────────────────► [Экран корзины]
    │                                          │
    │                                          ├─ Оформить заказ ─► [Checkout: шаги 1–4]
    │                                          ├─ Очистить корзину ─► (очистка, остаёмся или → Главное меню)
    │                                          ├─ Удалить товар / Изменить кол-во (на месте)
    │                                          └─ Назад ─► [Главное меню]
    │
    └─ 📦 Мои заказы ───────────────────► [Список заказов]
                                               │
                                               ├─ 1 заказ: кнопка «Открыть заказ» ─► [Экран заказа #ID]
                                               ├─ Несколько: клик по строке #ID ─► [Экран заказа #ID]
                                               └─ Назад ─► [Главное меню]

[Экран заказа #ID]
    ├─ Написать мастеру ─► (внешняя ссылка / deep link в Telegram)
    └─ Назад ─► [Список заказов]

[После оформления заказа]
    Сообщение: «Ваш заказ №XXXX оформлен.» + кнопка «Посмотреть заказ» ─► [Экран заказа #ID]

[При изменении статуса заказа]
    Уведомление (текст по статусу) + кнопка «Посмотреть заказ» ─► [Экран заказа #ID]
```

---

## 4. Структура экранов (UI specification)

### 4.1. Экран корзины

**ID экрана:** `cart`.

**Заголовок:** «Корзина» (при необходимости — «Корзина (N)» в шапке).

**Содержимое:**

1. **Список товаров** (если пусто — текст «Корзина пуста»):
   - Название товара
   - Цена (за единицу)
   - Опции: цвет кожи, фурнитура
   - **Количество:** кнопки «−» и «+», между ними число
   - Кнопки: **Изменить** (опционально), **Удалить** (удаление позиции)

2. **Итог:**
   - Строка: «Итого: X XXX ₽» (сумма по товарам, без доставки).

3. **Кнопки действий:**
   - **Оформить заказ** (primary) → переход в сценарий checkout (шаг 1).
   - **Очистить корзину** (secondary) → очистка корзины, подтверждение при необходимости.
   - **Главное меню** или **Назад** (secondary) → возврат в main-menu.

**Данные:**
- Список позиций корзины: `[{ product, color, hardware, quantity }]`.
- Итог считается на фронте/боте как сумма (price × quantity) по позициям.

**Переходы:**
- Оформить заказ → `checkout`, шаг 1.
- Назад / Главное меню → `main-menu`.

---

### 4.2. Экран «Мои заказы» (список заказов)

**ID экрана:** `order-list`.

**Заголовок:** «Мои заказы».

**Вариант А — один заказ:**

- Блок карточки:
  - **Заказ #1042**
  - **Статус:** В работе
  - **Дата:** 14 марта
  - **Сумма:** 3 200 ₽
- Кнопка: **Открыть заказ** → экран заказа #1042.

**Вариант Б — несколько заказов:**

- Список строк (каждая — один заказ):
  - `#1042 — В работе`
  - `#1038 — Отправлен`
  - `#1032 — Завершен`
- Клик по строке (или отдельная кнопка «Открыть») → экран выбранного заказа.

Внизу экрана: кнопка **Назад** → главное меню.

**Данные:**
- Список заказов пользователя: `[{ id, status, date, totalAmount }]`.
- Сортировка: сначала новые (по дате создания или обновления).

**Переходы:**
- Выбор заказа → экран заказа `order-detail` с `orderId`.
- Назад → `main-menu`.

---

### 4.3. Экран заказа (детали)

**ID экрана:** `order-detail`. Параметр: `orderId`.

**Заголовок:** «Заказ #ID» (например, «Заказ #1042»).

**Содержимое (блоки сверху вниз):**

1. **Статус заказа**  
   Одна строка: текущий статус (из списка статусов ниже).

2. **Состав заказа**  
   Список позиций: название, цвет/фурнитура, кол-во, цена, сумма по позиции.

3. **ПВЗ (пункт выдачи)**  
   Название и адрес пункта выдачи (одним блоком).

4. **Комментарий клиента**  
   Текст комментария или «—», если пусто.

5. **Стоимость:**
   - Стоимость товаров: X XXX ₽
   - Стоимость доставки: X XXX ₽ • срок доставки Y–Z дней (СДЭК)
   - **Итоговая стоимость:** X XXX ₽

**Кнопки:**
- **Написать мастеру** — переход в чат/контакт мастера (ссылка или `tg://`).
- **Назад** — возврат на экран «Мои заказы».

**Данные:**
- `orderId`
- Полный объект заказа: статус, состав, адрес, комментарий, суммы (товары, доставка, итог).

**Переходы:**
- Назад → `order-list`.
- Написать мастеру → внешнее действие (не смена экрана бота).

---

## 5. Статусы заказа

Используются в списке заказов и на экране заказа. Отображаются как есть (без кода в UI).

| Код (внутр.) | Отображаемое название |
|--------------|------------------------|
| `pending` | Ожидает подтверждения |
| `clarification` | Требует уточнения |
| `confirmed` | Подтвержден |
| `paid` | Оплачен |
| `in_progress` | В работе |
| `shipped` | Отправлен |
| `completed` | Завершен |
| `rejected` | Отклонен |

## 6. Сообщения и уведомления бота

### 6.1. После оформления заказа

**Момент:** пользователь на последнем шаге checkout нажал «Подтвердить заказ», заказ создан в системе.

**Текст сообщения:**
```
Ваш заказ №XXXX оформлен.
```

**Inline-кнопка (одна):**
- Текст: **Посмотреть заказ**
- Действие: открыть экран заказа с `orderId` только что созданного заказа (или отправить сообщение с той же кнопкой, по нажатию — переход на экран заказа).

### 6.2. При изменении статуса заказа

**Момент:** мастер/система изменила статус заказа (подтверждён, оплачен, в работе, отправлен и т.д.).

**Текст сообщения:** короткое уведомление, например:
- «Статус заказа №XXXX изменён: Подтвержден.»
- «Заказ №XXXX отправлен.»
- «Заказ №XXXX завершён. Спасибо!»

**Inline-кнопка (одна):**
- Текст: **Посмотреть заказ**
- Действие: открыть экран заказа с соответствующим `orderId`.

## 7. Структура кнопок (Inline Keyboard, Telegram)

Ниже — логическая структура; в Telegram это реализуется как `InlineKeyboardMarkup`.

### 7.1. Главное меню — нижняя панель

- **Сценарий 2:** один ряд, одна кнопка  
  `[ 🛒 Корзина (N) ]`  
  callback_data: `cart_open` или `screen:cart`

- **Сценарий 3:** один ряд, одна кнопка  
  `[ 📦 Мои заказы ]`  
  callback_data: `orders_list` или `screen:order-list`

- **Сценарий 4:** один ряд, две кнопки  
  `[ 🛒 Корзина (N) ] [ 📦 Мои заказы ]`  
  callback_data: `cart_open` | `orders_list`

### 7.2. Экран корзины

- Кнопки под каждой позицией: изменение количества, удаление (при необходимости — через callback_data с `cart_item_id` или индексом).
- Внизу экрана (reply keyboard или inline, по выбору реализации):
  - `[ Оформить заказ ]` — `checkout_start`
  - `[ Очистить корзину ]` — `cart_clear`
  - `[ Назад ]` — `screen:main-menu`

### 7.3. Список заказов

- Один заказ: одна кнопка  
  `[ Открыть заказ ]` — `order_open:{orderId}`

- Несколько заказов: список из кнопок (или одна кнопка на заказ)  
  `[ #1042 — В работе ]` — `order_open:1042`  
  `[ #1038 — Отправлен ]` — `order_open:1038`  
  и т.д.
- `[ Назад ]` — `screen:main-menu`

### 7.4. Экран заказа

- `[ Написать мастеру ]` — `contact_master` или URL (например, t.me/…).
- `[ Назад ]` — `screen:order-list`

### 7.5. Сообщение «Заказ оформлен» и уведомления о статусе

- Одна кнопка под сообщением:  
  `[ Посмотреть заказ ]` — `order_open:{orderId}`

---

## 8. Данные, передаваемые между экранами

| Откуда | Куда | Данные |
|--------|------|--------|
| main-menu | cart | — (корзина берётся из состояния/сессии пользователя) |
| main-menu | order-list | — (список заказов запрашивается по user_id) |
| order-list | order-detail | `orderId` (обязательно) |
| order-detail | order-list | — (при «Назад») |
| cart | checkout | Текущая корзина (передаётся в шаг 1 checkout) |
| checkout (успех) | — | Созданный `orderId`; бот отправляет сообщение + кнопку «Посмотреть заказ» с этим `orderId` |
| Уведомление о статусе | order-detail | `orderId` в callback_data кнопки «Посмотреть заказ» |

**Состояние/сессия пользователя:**
- Корзина: массив позиций (product, color, hardware, quantity), привязанный к user_id (или session).
- Текущий экран (и при необходимости история) для кнопки «Назад».

**Сервер/БД:**
- Заказы пользователя: список с полями id, status, date, totalAmount, и при открытии заказа — полный объект заказа (состав, адрес, комментарий, суммы).

## 9. Краткая матрица экранов

| Экран | Заголовок | Ключевые элементы | Исходящие переходы |
|-------|-----------|-------------------|---------------------|
| main-menu | — | 6 кнопок раздела + динамическая панель (Корзина / Мои заказы) | catalog, collections, new-items, gallery, about, cart, order-list |
| cart | Корзина | Список позиций, кол-во, итог, Оформить / Очистить / Назад | checkout, main-menu |
| order-list | Мои заказы | Список заказов или одна карточка, «Открыть заказ», Назад | order-detail, main-menu |
| order-detail | Заказ #ID | Статус, состав, адрес, комментарий, суммы, Написать мастеру, Назад | order-list, внешняя ссылка |

## 10. Примечания по реализации

- **Динамическая панель:** при каждом отображении главного меню проверять: (1) корзина не пуста, (2) у пользователя есть хотя бы один заказ. В зависимости от этого показывать 0, 1 или 2 кнопки.
- **«Мои заказы»:** если заказов нет, кнопку «Мои заказы» в панели не показывать (сценарии 1 и 2).
- **Экран заказа:** при переходе по «Посмотреть заказ» из сообщения/уведомления можно открывать сразу `order-detail` с переданным `orderId`, без обязательного захода в order-list.
- Тексты кнопок и статусов в документе даны как финальные для UI; внутренние коды статусов (pending, in_progress и т.д.) — для бэкенда и callback_data при необходимости.

## 11. Post-order flow и доставка СДЭК

Все экраны после оформления заказа (post-order-demo, order-detail, post-order-confirmed, post-order-clarification, post-order-in-progress, post-order-shipped, post-order-rejected, post-order-payment-received, order-cancel-confirm, order-cancelled-auto, мастерские post-order-new, master-order-paid, master-order-in-production) должны показывать доставку в едином формате:

- **Доставка:** СДЭК, ПВЗ: название, адрес ПВЗ, стоимость доставки, срок доставки.
- **Итого:** сумма заказа с учётом доставки.

Не показывать: «Адрес доставки», `delivery = "—"`, тексты про расчёт доставки позже. При отсутствии данных использовать подпись «уточняется». Кнопка контакта: **Написать нам** (единообразно). При наличии `activeOrder` демо-сценарии не перезаписывают его, а используют как источник данных.
